# v0.6 — Adversarial review

> Status: in progress
> Branch: `v0.6-redesign`
> Target: `theroundtaible@0.6.0` on npm when verified end-to-end

## Thesis

The current `discuss` command optimizes for the wrong thing. Knights see each other's responses, know they need a 9/10 score to "win", and converge on polite agreement. The output is three essays the user still has to summarize.

The thesis of this redesign:

> The product of a multi-LLM tool is **disagreement that resolves into signal**, not consensus. We extract value from how the models differ, not from making them agree.

## The new pipeline

Three explicit roles per question. Each role is a single LLM call. No rounds. No scores.

```
                                              ┌──────────────┐
        ┌──────────┐                          │              │
input ─►│ Proposer │ ──── proposal ──────────►│              │
        └──────────┘                          │              │
                                              │   Critic     │ ── strongest
                                              │              │    objection
                                              │              │
                                              └──────┬───────┘
                                                     │
                                                     ▼
                                              ┌──────────────┐
                                              │              │
                                              │ Synthesizer  │ ── decision record
                                              │              │
                                              └──────────────┘
```

- **Proposer** — given the topic and project context, writes a concrete proposal with reasoning. ~300 words. Returns structured fields: `recommendation`, `why`, `risks`.
- **Critic** — given the topic, project context, and the proposer's full output, must produce the *single strongest* objection. Not a list. One knockout punch. Returns: `strongest_objection`, `severity` (low/medium/high), `would_change_recommendation` (bool).
- **Synthesizer** — given everything above, writes the final decision record. Acknowledges the objection. States whether it changes the recommendation. Reports a confidence level. Flags if the disagreement was suspiciously low ("knights agreed too easily — possible shared blindspot").

## CLI

New command, parallel to `discuss`. `discuss` stays unchanged in v0.6 — it is deprecated in v0.7 if `advise` is the better tool.

```
roundtable advise "<question>"
```

Optional flags:
- `--proposer <name>` — override role assignment
- `--critic <name>`
- `--synth <name>`
- `--no-context` — skip project context, use topic only
- `--json` — machine-readable output for scripting

If roles are not provided, they are picked from `config.knights[].capabilities` based on simple keyword matching against the topic. Fallback: priority order (lowest priority = proposer, etc.).

## Output format

Replaces the transcript-style `decisions.md`. Written to `.roundtable/sessions/<timestamp>-<slug>/decision.md`:

```markdown
# Decision: <topic>

**Date:** YYYY-MM-DD
**Proposer:** <knight>
**Critic:** <knight>
**Synthesizer:** <knight>
**Confidence:** low | medium | high
**Disagreement health:** healthy | suspicious-agreement | unresolved-conflict

## Recommendation

<one paragraph from synth>

## Reasoning

<bullets from proposer, condensed by synth>

## Strongest counter-argument

<critic's knockout, verbatim>

**Would this change the recommendation?** <yes/no, from synth>
**If yes, what changes:** <one paragraph from synth>

## Open questions

- <bullets, only the ones the synth couldn't resolve>

---

## Appendix — full role outputs

(collapsed by default in any reader; raw text from each role for audit)
```

That is a decision document. Scannable in 30 seconds. Auditable when needed.

## Smoke-test evidence

### Run 1 — first end-to-end (2026-04-28)

- **Topic:** "Should TheRoundtAIble drop the chronicle.md file in favor of a curated principles.md, or is the auto-appended chronicle still load-bearing?"
- **Pipeline:** Claude (proposer) → Gemini (critic) → GPT (synthesizer)
- **Outcome:** Confidence high. Disagreement health: healthy. Critic raised a substantive high-severity objection that flipped to "yes, would change recommendation". Synthesizer kept the recommendation but explicitly modified the migration plan in response.

The decision record exposed a genuine v0.7 candidate: replace the `chronicle.md` auto-append with curated `principles.md` plus a one-shot migration helper. That finding is itself the strongest evidence that the new pipeline works — it surfaced a concrete next step that `discuss` would have buried in an essay.

### Run 2 — head-to-head A/B vs `discuss` (2026-04-28)

Same question fed to both commands, codebase scan disabled in `discuss` for fair comparison:

- **Topic:** "Should TheRoundtAIble's advise command default to including project source code in its context, or stay opt-in via a --source flag?"

|                       | `discuss`                                  | `advise`                                       |
|-----------------------|--------------------------------------------|------------------------------------------------|
| Wall-clock            | 265 s (2 rounds × 3 knights = 6 calls)     | 97 s (3 calls, sequential)                     |
| Output bytes          | 31 KB (decisions.md + discussion.md)       | 8.7 KB (single decision.md)                    |
| Recommendation        | Add new `--context auto|none|full` flag, replace boolean readSourceCode, use git ls-files, dynamic spinner, 8-file scope | Keep `--source` opt-in, strengthen the *default* context bundle, emit a hint when context is insufficient |
| Implementation detail | High — 6 numbered steps, file paths        | Lower — recommendation level, no file list     |
| Critic engagement     | Gradual convergence to polite 9/10 consensus | Single high-severity objection from Gemini, synthesizer modified the implementation bar but kept the recommendation |

**Honest verdict.** For a *decision* question — which is what `advise` is designed for — `advise` produced a cleaner, faster, lower-risk answer. The synthesizer's "the objection changes the implementation bar, not the recommendation" is exactly the kind of synthesis the redesign was built for.

`discuss` was not useless. It produced a more concrete implementation plan (Gemini's six-step list) which `advise` did not. If the user's question is "how do I implement this", `discuss` is currently the better tool. If the question is "should I do this at all and which way", `advise` wins.

This points at the v0.7 shape from the roadmap: **modes per use case**. `advise` for decisions, a future `plan <topic>` for implementation breakdown, drop the one-size-fits-all `discuss`.

**Important meta-finding.** The two commands gave *different recommendations* on the same input. `discuss` proposed adding a new mode system. `advise` proposed keeping the existing flag and improving defaults. Both are defensible, but the divergence shows that `discuss`'s "9.3/10 consensus" is not delivering reliable signal — it's converging on something, but not on the same thing across runs or commands. That is consistent with the original critique that score-based consensus is theater.

### Verdict at this checkpoint

`advise` clears the bar. Continue with v0.7 planning. The redesign branch was the right call.

## Open design questions

- Should the critic see only the proposer's output, or also project context? Bias risk vs grounding tradeoff. **Default for first cut:** critic sees both. We can tighten later.
- Should the synthesizer be a different LLM than the proposer and critic? Probably yes — it should be neutral. Enforce in role assignment or let it overlap?
- Capability matching is fragile. For first cut, fall back to priority order: lowest priority = proposer, middle = critic, highest = synthesizer. Improve in v0.7.

## Out of scope for v0.6

- Removing `discuss`
- Replacing chronicle
- New commands beyond `advise`
- Cost preview
- Disagreement detection beyond a simple synth flag

These are v0.7+ work. Resist scope creep here. The point of v0.6 is to prove the role-based pipeline is better — nothing else.
